Core Bedrock agent services
Orchestration
Bedrock Agents
ReAct-loop orchestrator. Action groups → Lambda. Session memory.
Use when: you need an agent that autonomously plans, calls tools, and loops until task complete
RAG
Knowledge Bases
Managed RAG: ingest → chunk → embed → retrieve
Use when: agent needs to answer from your private docs without hallucination
Safety
Guardrails
Content filters, PII, topic blocks, grounding checks
Use when: deploying to end-users or regulated industries; mandatory for compliance
Workflow
Bedrock Flows
Visual prompt chaining: conditions, loops, Lambda nodes
Use when: workflow is linear/conditional and you want visual authoring over custom code
Dynamic
Inline Agents
Runtime-defined agents — instructions + tools per call
Use when: each request needs a different agent config (multi-tenant, per-user tools)
Multi-agent
Multi-Agent
Supervisor → specialist sub-agents with own tools/KBs
Use when: single agent is too broad; decompose into specialist agents with clear boundaries
Memory & context management
Short-term
Session memory
Auto-managed per sessionId. Multi-turn context in one session.
Use when: conversation is within one session (e.g. support chat)
Long-term
Agent memory
Cross-session summaries stored and retrieved per user
Use when: users return across sessions and need continuity
Custom
External memory
DynamoDB / ElastiCache via Lambda action group
Use when: you need structured/auditable memory or high volume state

Action execution & tooling
Actions
Action Groups
OpenAPI schema → Lambda. Primary tool-call mechanism.
Default for any action the agent needs to take in the world
Compute
Code Interpreter
Agent writes + runs Python in sandbox. Returns results.
Use when: agent needs to compute, analyse data, or generate charts dynamically
Orchestration
Step Functions
Durable workflows, retries, parallel branches + Bedrock
Use when: workflow is long-running, needs retry logic or must fan-out in parallel
Monitoring & observability
Metrics
CloudWatch
Latency, token usage, error rate metrics & alarms
Default monitoring. Always enable for production.
Logging
Invocation Logging
Full prompt + response + agent trace logs to S3/CW
Enable in production: needed for audit, debugging agent traces
Tracing
X-Ray
Distributed traces across Lambda, SFN, Bedrock calls
Use when: debugging latency or tracing requests across multiple AWS services
